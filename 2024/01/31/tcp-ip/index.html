TCP/IP模型

TCP/IP协议模型，包含了一系列构成互联网基础的网络协议，是Internet的核心协议。

基于TCP/IP的参考模型将协议分成四个层次，它们分别是链路层、网络层、传输层和应用层。下图表示TCP/IP模型与OSI模型各层的对照关系。

TCP/IP协议族按照层次由上到下，层层包装。最上面的是应用层，这里面有http，ftp,等等我们熟悉的协议。而第二层则是传输层，著名的TCP和UDP协议就在这个层次。第三层是网络层，IP协议就在这里，它负责对数据加上IP地址和其他的数据以确定传输的目标。第四层是数据链路层，这个层次为待传送的数据加入一个以太网协议头，并进行CRC编码，为最后的数据传输做准备。

  • TCP协议是传输层协议，而IP协议是网络层协议
  • TCP是可靠的传输协议，ip是不可靠的网络层协议（只保证快）

发送和接收报文过程

客户端发送http数据包

  1. 应用层到传输层：数据包增加TCP头部
  2. 传输层到网络层：增加IP头部
  3. 网络层到网络接口层（网络访问层）：增加以太网头部
  4. 最后通过物理层传输到服务端

服务端接收http数据包

依次去掉以太网头部、ip头部和tcp头部

数据包的封装（发送端）与解封装（接收端）的过程

IP地址

  • IP地址是IP协议提供的一种统一的地址格式，它为互联网上的每一个网络和每一台主机分配一个逻辑地址，以此来屏蔽物理地址的差异。可以说每一台电脑都有唯一的ip地址

  • IP地址由网络号和主机号两部分组成。其中网络号用于标识一个网络，主机号用于标识该网络中的一个特定主机。（分层结构）ip地址在整个互联网范围内唯一。

  • ip地址可以映射到硬件地址，但不是简单的映射关系。地址解析协议ARP解决这个问题的方法是在主机ARP高速缓存中存放一个从IP地址到硬件地址的映射表，

IPV4

IPv4使用32位地址，通常被写作点分十进制的形式，即每8位分成一段，每个8位的段用十进制表示，（84=32）中间用点分隔。*例如：11000000. 10101000.00000001.00000001就表示为192.168.1.1

IPV6

IPv6使用128位地址，采用冒号十六进制的表示方法，即每16位分成一段，每个16位的段用十六进制表示（16*8=128），中间用冒号分隔。

TCP三次握手和四次挥手

三次握手

建立一个 TCP 连接需要“三次握手”（发送三次报文）

  • 一次握手:客户端发送带有 SYN（SEQ=x） 标志的数据包 -> 服务端，然后客户端进入 SYN_SEND 状态，等待服务器的确认；
  • 二次握手:服务端发送带有 SYN+ACK(SEQ=y,ACK=x+1) 标志的数据包 –> 客户端,然后服务端进入 SYN_RECV 状态
  • 三次握手:客户端发送带有 ACK(ACK=y+1) 标志的数据包 –> 服务端，然后客户端和服务器端都进入ESTABLISHED 状态，完成 TCP 三次握手。

当建立了 3 次握手之后，客户端和服务端就可以传输数据了。

为什么不能是两次握手

  • 主要因为tcp是一个可靠的传输协议，如果只使用两次握手来建立TCP连接，那么客户端可能会发出虚假的半连接，即一个没有经过服务器确认的连接请求。这会导致服务器等待一个永远不会到来的确认，从而造成资源浪费。

  • 客户端需要对服务端的SYN进行ACK确认

四次挥手

断开一个 TCP 连接则需要“四次挥手”：

  1. 第一次挥手：客户端发送一个 FIN（SEQ=x） 标志的数据包->服务端，用来关闭客户端到服务器的数据传送。然后客户端进入 FIN-WAIT-1 状态。
  2. 第二次挥手：服务器收到这个 FIN（SEQ=X） 标志的数据包，服务端发送一个 ACK （ACK=x+1）标志的数据包->客户端 。然后服务端进入 CLOSE-WAIT 状态，客户端进入 FIN-WAIT-2 状态。
  3. 第三次挥手：服务端发送一个 FIN (SEQ=y)标志的数据包->客户端，请求关闭连接，然后服务端进入 LAST-ACK 状态。
  4. 第四次挥手：客户端发送 ACK (ACK=y+1)标志的数据包->服务端，然后客户端进入TIME-WAIT状态，服务端在收到 ACK (ACK=y+1)标志的数据包后进入 CLOSE 状态。此时如果客户端等待2MSL 后依然没有收到回复，就证明服务端已正常关闭，随后客户端也可以关闭连接了。

只要四次挥手没有结束，客户端和服务端就可以继续传输数据

为什么不能把服务器发送的 ACK 和 FIN 合并起来，变成三次挥手？

  • 服务端接收到客户端的FIN数据包后会立马发送ACK确认（收到了），然后过一段时间（处理剩余的请求，释放资源）才会发送FIN。如果将ACK和FIN合并成第二次挥手，那么客户端等待第二次挥手的时间就比较久，客户端会认为服务器端没有收到FIN而不断重发

为什么第四次挥手客户端需要等待 2*MSL（报文段最长寿命）时间后才进入 CLOSED 状态？

  • 为了确认服务端是否收到ACK包，如果服务器在2MSL内没有收到ACK就会重发FIN给客户端。（保证可靠性）

总结

  • 三次握手:1、我要传输 2、允许传输 3、即将开始
  • 四次挥手:1、我要走了 2、我知道了你先等我把剩下工作干完 3、干完了可以断开了 4、好的
  • SYN用于三次握手，FIN用于四次挥手，二者里面的seq=x称为序列号
  • SYN/FIN 不包含数据但要消耗一个序列号
  • 凡是需要对端确认的，一定消耗tcp报文的序列号

半连接队列和SYN Flood攻击

客户端大量伪造IP 发送SYN 包，服务端回复的 ACK+SYN 去到了一个[末知]的IP 地址，势必会造成服务端大量的连接处于 SYN_RCVD 状态（等待第三次握手）即产生大量的半连接。而服务器的半连接队列大小也是有限的，如
果半连接队列满，也会出现无法处理正常请求的情况。这种攻击就叫做SYN Flood攻击（SYN 洪水）

TFO（TCP快速打开）原理

TFO(TCP Fast Open)是在原来TCP协议上的扩展协议，它的主要原理就在发送第一个SYN包的时候就开始传数据了，不过它要求当前客户端之前已经完成过「正常」的三次握手。

快速打开分两个阶段:请求Fast Open Cookie和真正开始TCP Fast Open

Fast Open Cookie

  • Cookie在网络中通常用于存储用户信息，以实现网站对用户的个性化服务，例如记住用户名、记录浏览历史等。

TCP Fast Open

优势

  • 可以利用握手去除一个往返的RTT
  • RTT表示从发送端发送数据开始，到发送端收到来自接收端的确认（接收端收到数据后便立即发送确认），总共经历的时延。
  • 可以防止SYN-Flood之类的攻击

TCP报文中的时间戳作用

TCP Timestamps Option由四部分组成：

类别、长度。发送方时间戳和回显时间戳

计算往返时延RTT

在启用Timestamps选项后，因为ACK包里包含了TSval和TSecr，这样无论是正常确认包，还是重传确认包，都可以通过这两个值计算出RTT

防止序列号的回绕问题

假设发送了6个数据包，每个数据包的大小为1GB，第五个包序列号 发生回绕。第二个包因为某些延迟原因导致重传，但没有丢失，到t7时刻才到达。

这个迷途数据包与后面要发送的第6个包序列号完全相同，如果没有一些措施进行区分，将会造成数据的紊乱。

有TimeStamps的存在，第2个和第6个发送时间戳不一样，就能够区分

TCP超时重传时间（RTO）

RTO（Retransmission Timeout）超时重传时间

TCP具有超时重传时间机制，即发送数据包后间隔一段时间没有等到数据包的回复时，重传这个数据包

  • RTO设置的太小会导致不必要的重传
  • RTO设置太大，有些包丢失了很久才重传

设置RTO的经典方法（适用于RTT波动较小的情况）：

简单思想：取平均值，第一次RTT为400ms，第二次为800ms，那么第三次发送时就设置RTO为600ms

  • 核心：RTO会根据上次传输的RTT进行更新
  • 引入了平滑往返时间

TCP流量控制和TCP拥塞控制

TCP流量控制

对于发送端和接收端而言，TCP需要把发送的数据放到发送缓冲区，把接收到的数据放到接收缓冲区

流量控制目的是防止分组丢失，就是通过接收缓冲区的大小，控制发送端的发送（通过滑动窗口大小来实现）。当接收端的数据处理速度跟不上发送端的速度时，接收端会将滑动窗口大小置0（在ACK确认的时候在TCP头部有个窗口大小的字段），然后发送端暂时就不能发送数据了。

  • 分组丢失是指在网络传输过程中，由于各种原因导致分组数据包无法到达目的地，从而造成分组数据丢失的现象。
  • 滑动窗口由发送窗口和接收窗口构成，但滑动窗口大小不一定等于发送窗口大小加上接收窗口大小。

发送窗口：

发送端数据包的四种状态：

  • 已发送且已确认（ACK确认）
  • 已发送未确认
  • 未发送但可以发送（接收端有空间可以接收）
  • 未发送且不可以发送（接收端没空间接收）

接收窗口：

接收端数据包的三种状态：

  • 已接收
  • 未接收且准备接收
  • 未接收并且未准备接收

接收窗口：接收缓冲区中空闲的部分，里面是未接收且准备接收的数据

  • 右边部分表示的是已经接收但没有处理完的数据，

发送端发送数据包后，接收端进行ACK确认的时候会带上滑动窗口的大小。

TCP拥塞控制

拥塞控制是防止过多的数据注入到网络中，避免出现网络负载过大的情况；常用的方法就是：（ 1 ）慢开始、拥塞避免（ 2 ）快重传、快恢复

区别

  • 拥塞控制作用于网络，流量控制作用于接收端

TCP的kepp-alive原理

  • 客户端和用户端建立TCP连接后，客户端有可能故障宕机或者断网下线，而服务端一无所知。导致了死连接

  • keepalive机制就是通过定时发送探测包来探测连接的对端是否存活，默认时间是7200s(两小时)

  • 如对于上述情况，每隔两小时服务端就会发送一个探测包给客户端来探测连接是否仍然有效。

TCP端口号（port）

端口号位于传输层数据包的TCP头部，且有两部分：源端口和目标端口，比如我的电脑要访问百度

我的电脑会起一个临时端口作为源端口，发送到百度，需要百度端口（80）作为目标端口。（两个端口对接了就访问到了百度）

熟知端口号：范围0~1023

  • HTPP：80
  • HTTPS：443
  • SSH：22

已登记的端口号：范围1024~49151

  • Mysql： 3306
  • Redis：6379
  • MongoDB： 27017

临时端口号：范围49152~65535

确认号为：500+300+200 = 1000

telnet的用法

检查端口号是否打开

telent [domainname or ip][port]这条命令能告诉我们到远端server指定端口的网连接是否可达。例如在命令行中输入：

telnet www.baidu.com 80

可以测试我的电脑是否可以和百度进行通信。

  • 在Windows系统中，默认情况下Telnet功能是关闭的，需要手动开启。开启方法如下：
    1. 打开控制面板，选择”程序”。
    2. 点击”启用或关闭Windows功能”。
    3. 勾选”Telnet客户端”，点击”确定”进行安装。
    4. 安装完成后重启计算机生效。

telnet发送HTTP请求

TCP和UDP的区别

TCP UDP
是否连接 面向连接（三次握手） 无连接（可直接发送数据）
是否可靠 可靠传输，使用流量控制和拥塞控制 不可靠传输，不使用流量控制和拥塞控制
连接对象个数 只能一对一通信 支持一对一、一对多、多对一和多对多交互通信
数据传输方式 面向字节流 面向报文
首部开销 首部最小20字节，最大60字节 首部开销小，仅8字节
适用场景 要求可靠传输的应用例如文件传输 适用于实时应用如视频会议、直播

TCP可靠性体现：

  1. 确认响应：对于一个收到的请求，将发送一个确认。
  2. 超时重传：当TCP发出一个段后，它启动一个定时器，等待目的端确认收到这个报文段。如果不能及时收到一个确认，将重发这个报文段。
  3. 数据编号：TCP给发送的每一个包进行编号，接收方对数据包进行排序，把有序数据传送给应用层。
  4. 校验和：TCP将保持它首部和数据的检验和。这是一个端到端的检验和，目的是检测数据在传输过程中的任何变化。
  5. 数据去重：TCP的接收端会丢弃重复的数据。
  6. 流量控制和拥塞控制也能体现

面向字节流：

  1. 连续传输：数据在传输过程中被看作是字节的连续流。发送端和接收端之间没有明确的边界或消息边界。
  2. 流式通信：这种模型中，发送端连续发送数据，接收端也连续接收数据。通信的双方都认为数据是一个持续不断的数据流。
  3. 字节编号：每个字节都有一个特定的编号，这样接收端可以按顺序重新组装数据。
  4. 缓冲和流量控制：由于没有明确的消息边界，发送端可能需要缓冲数据，并使用流量控制机制来确保接收端能够跟上发送端的速率。

面向报文：

  1. 消息导向：在这种模型中，数据被组织成独立的消息或数据包，每个消息都有明确的边界。
  2. 消息传输：发送端发送一个完整的消息，然后等待接收端的确认。一旦收到确认，发送端就可以继续发送下一个消息。
  3. 消息优先级和路由：面向报文的协议通常支持消息优先级和路由功能，允许根据不同的策略处理和转发消息。
  4. 错误检测和重传：由于消息之间有明确的边界，所以可以在接收端检测到错误或丢失的消息，并请求重传。
  5. 简单和高效：这种模型通常更简单、高效，适用于那些不需要流式通信的场景。

这两种方式各有优缺点，选择使用哪种方式取决于特定的应用场景和需求。例如，HTTP协议是一个典型的面向字节流的协议，而SMTP（简单邮件传输协议）是一个典型的面向报文的协议。
