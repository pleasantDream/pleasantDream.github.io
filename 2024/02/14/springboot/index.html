在application.properties配置文件中修改端口号和虚拟目录

server.port=9090     # 用.来体现层级
server.servlet.context-path=/start

访问的url就修改为： http://localhost:9090/start/hello

### Bean注册

![](./springboot/009.png)

- 如果要注册的Bean对象来自第三方（不是自定义的比如maven提供的类），无法用@Component及其衍生注解来注册Bean的。只能用**@Bean或@import**来注册Bean对象

#### @Bean

- 在启动类里面添加一个方法来注册Bean对象如：

```java
@Bean  
// 注入Country对象,将方法返回值交给ioc容器
// Bean对象的名字默认是方法名，这里是country
// 要修改的话可以@Bean("名字")
public Country country(){
	return new Country();
}

  • 如果方法的内部需要用到ioc容器中已经存在的Bean对象，只需在方法上声明，spring会自动注入如：

@Bean  
public Province province(Country country){
	return new Province();
}
  • 不建议在启动类里面添加一个方法来注册Bean对象，因为这样会让启动类臃肿（启动类就干一个启动的活就够了），建议新建一个配置类，在配置类里面集中注册如：
@Configuration
//该类需要放到启动类的包及其子包下，这样启动类才能扫描到该配置类
public class CommonConfig{ 
	@Bean
	public Country country(){
	return new Country();
}}

@import

  • 导入配置类，如：
@Import({xxx.class}) //其中xxx是配置类的类名
  • 当配置类比较多的时候可以导入ImportSelector接口实现类如：
@Import(xxx.class) //注意没有{}，其中xxx是实现类的类名
// 在接口实现类里面有全部类名的数组
// 实际上就是把多个@Import注解放到接口实现类里面，让启动类上的@Import注解只有一个，显得简洁优美
  • @EnableXxxx注解，封装@Import注解（类似@SpringBootApplication的组合注解）

从配置文件中获取参数

  • 前导：在声明一个bean对象的时候可以给bean对象的一些属性赋值

@Value ： 作用，从配置文件中读取键名对应的值如：

   //如果配置文件中配置了指定的信息,则注入,否则不注入
   @ConditionalOnProperty(prefix = "country",name = {"name","system"})  	
@Bean
   // 自动读取yml配置文件中的键名对应的值给name参数和system参数
   public Country country(@Value("${Country.name}") String name, @Value("${Country.system}") String system){
       Country country = new Country();
       country.setName(name);      // 给name属性赋值
       country.setSystem(system); // 给system属性赋值
       return country ;
   }

相关概念

注册Bean对象是Spring框架中的概念，指的是将一个对象添加到Spring应用上下文中，使其可以被其他对象通过依赖注入等方式使用。

声明Bean对象通常是指在使用XML配置方式时，在配置文件中声明一个Bean，指定其名称、作用域、依赖关系等信息。通过声明Bean，可以在Spring容器中创建相应的对象实例，并管理其生命周期。

springboot自动配置原理（重要）

遵循约定大约配置的原则，在boot程序启动后，起步依赖中的一些bean对象会自动注入到IOC容器，不需要手动声明，简化开发。就是不用自己写配置类

以web起步依赖的自动配置为例：

如图，在spring-boot-auto-configure依赖下的配置文件org.springframework.boot.autoconfigure.AutoConfiguration.imports中有个全类名org.springframework.boot.autoconfigure.web.servlet.DispatcherServletAutoConfiguration

该类是一个组合注解，且该类有个注解@ConditionalOnClass（DispatcherServlet.class），表示当前环境中存在DispatcherServlet这个类时，才声明DispatcherServletAutoConfiguration这个Bean类，而引入了web起步依赖就会有DispatcherServlet这个类，总结一下就是： 引入了web起步依赖，相关配置类才会生效（由@ConditionalOnClass注解实现），没有引入web依赖，相关配置类就不会生效

总结：

核心在AutoConfiguration.imports文件里面，该配置文件里面有很多全类名

springboot提供自动配置类，将其全类名写入到AutoConfiguration.imports文件中，当引入一个依赖时，相关的配置类就会生效

多环境开发-profiles

springboot提供的profiles可以用来隔离应用程序配置的各个部分，并在特定环境下指定部分配置生效

如何分割不同环境的配置

用 ---分割

如何指定哪些配置属于哪个环境

spring.config.activate.on-profile=环境名称

如何指定哪个环境的配置生效

spring.profiles.actice= 环境名称

#通用信息，指定生效的环境
spring:
  profiles:
    active: dev
---     # 分隔符
#开发环境
spring:
  config:
    activate:
      on-profile: dev  # 环境名
server:
  port: 8081    # 该环境的配置信息
---
#测试环境
spring:
  config:
    activate:
      on-profile: test    # 环境名
server:
  port: 8082    # 该环境的配置信息
---
#生产环境
spring:
  config:
    activate:
      on-profile: pro  # 环境名
server:
  port: 8083    # 该环境的配置信息

当可能的环境很多时，所有的环境信息写在一个yml文件中，不利于维护。

可以采用多文件配置的方式

多文件配置

将每个环境的配置信息分别写在不同的yml文件中, 在主yml 文件中指定生效的环境并配置共性信息

若在一个环境中写入很多的配置信息也会不利于维护，可以考虑使用分组配置

分组

可以根据不同功能来进行分组

spring:
  profiles:
    active: dev   # 指定dev分组下的所有配置文件生效
    group:
     "dev": devServer,devDB,devSelf
     "test": testServer,testDB,testSelf
     "pro": proServer,proDB,proSelf

场景

在实际开发中，经常会定义一些公共组件，提供给各个项目团队使用，而在springboot的项目中，一般会将这些公共组件封装为springboot的starter。

起步依赖组成：

我们会在starter中引入autoconfigure，这样starter依赖被引用时，autoconfigure也会被引用。

举例

需求：自定义mybatis的starter

步骤

使用：在工程中引入dmybatis-spring-boot-starter依赖即可

小知识、小工具

前后端分离

  • 一种开发模式，就是前端代码由前端工作人员写、后端代码由后端工作人员写。分工明确，效率高。
  • 在独立完成的同时都需要遵循共同的接口文档。前后端通过接口进行数据的交换
  • 前后端分离开发后，工程结构也会发生变化，即前后端代码不会混在同一个maven工程中，而是分为前端工程和后端工程。

lombok小工具

  • lombok 在编译阶段，为实体类自动生成setter getter toString等方法

  • 需要在pom文件中引入该依赖 在实体类上添加注解如：

    <dependency>
  <groupId>org.projectlombok</groupId>
  <artifactId>lombok</artifactId>
</dependency>
    @Data      // 成员变量的setter  getter toString等
@NoArgsConstructor  // 无参构造方法
@AllArgsConstructor  // 全参构造方法

  • 然后点击maven下的Lifecycle下的compile，就能在target包下看到添加方法的实体类

开发流程

  • 明确需求 -> 阅读接口文档 -> 思路分析 -> 开发 -> 测试

思路分析如写一个注册功能：

  • controller(需要做什么操作) -> Service(操作的实现方法) -> Mapper(实现方法对数据库的映射)
  • 中途遇到报错大多数是调用的方法在后面写，那为什么不从mapper层写到controller层呢，因为从controller层到mapper层的逻辑更清晰（从需求的方向走）。

Spring Validation

  • 其是Spring提供的一个参数校验框架，使用预定义的注解完成参数校验

步骤如下：

  • 导入validation坐标**(通过坐标指定依赖项)**
  • 在参数上添加**@Pattern注解**，制定校验规则
public Result register(@Pattern(regexp = "^\\S{5,16}$") String username,@Pattern(regexp = "^\\S{5,16}$") String password){这里面是Controller类中注册方法的代码} // regexp = "正则表达式"
  • 在Controller类上添加@Validated注解
  • 在全局异常处理器中处理参数校验失败的异常

全局异常处理器即是在启动类所在的包或子包下自定义一个异常处理器来捕获异常，并自定义异常的处理方式

@RestControllerAdvice     // @RestControllerAdvice常用于全局异常的捕获处理和请求参数的增强。
public class GlobalExceptionHandler {
    @ExceptionHandler(Exception.class)   // 接收所有异常
    public Result handleException(Exception e){
        e.printStackTrace();   // 在控制台输出异常，方便调试
        return Result.error(StringUtils.hasLength(e.getMessage())? e.getMessage(): "操作失败");   // 异常中有消息封装就输出该消息，否则输出操作失败
    }
}

把校验项进行归类分组，在完成不同的功能的时候，校验指定组中的校验项

  1. 定义分组（在实体内部定义接口）

正则表达式

正则表达式通常缩写为regex或regexp，是一种用于描述和匹配文本模式的字符串。

一些常用的正则表达式符号和用法：

  • .：匹配任意字符（除了换行符）。
  • ^：匹配字符串的开头。
  • $：匹配字符串的结尾。
  • *：匹配前面的字符或组出现0次或多次。
  • +：匹配前面的字符或组出现1次或多次。
  • ?：匹配前面的字符或组出现0次或1次。
  • {n}：匹配前面的字符或组恰好出现n次。
  • {n,}：匹配前面的字符或组出现n次或多次。
  • {n,m}：匹配前面的字符或组出现n到m次。
  • \d：匹配任意数字（相当于[0-9]）。
  • \D：匹配任意非数字字符（相当于[^0-9]）。
  • \w：匹配任意字母、数字或下划线字符（相当于[a-zA-Z0-9_]）。
  • \W：匹配任意非字母、数字或下划线字符（相当于[^a-zA-Z0-9_]）。
  • [abc]：匹配方括号内的任意字符。
  • [^abc]：匹配除了方括号内字符之外的任意字符。
  • \s：匹配任意空白字符（包括空格、制表符、换页符等）。
  • \S：匹配任意非空白字符。

登录认证

登录认证是指在应用中确认用户身份的过程。若没有登录认证，则用户可以在未登录的情况下（通过输入url绕过登录界面）访问到其他资源，很不安全。

  • web登录认证中最常用的规范是jwt令牌，令牌就是一段字符串

令牌作用

承载业务数据，减少后续请求查询数据库的次数

防篡改功能，保证信息的合法性和有效性

JWT

简介
  • 全称: JSON Web Token， 即用于web领域的基于JSON格式的令牌
  • 定义了一种简洁的、自包含的格式，用于通信双方以JSON数据格式安全地传输信息
  • 组成：
    • 第一部分:Header(头), 记录令牌类型，签名算法等。例如{“alg”：“HS256”， “type”：“JWT“}
    • 第二部分: Payload（有效载荷），携带一些自定义信息、默认信息等。例如{“id”: “1”, “username”: “Tom”}。 注：在此部分不要存放私密数据。因为这部分本身未加密
    • Signature（签名），为了确保数据没有被篡改，JWT标准要求在Header和Payload之间有一个签名。这个签名是通过将Header、Payload和密钥（通常在JWT的Header中指定）一起使用一个安全的（算法）哈希函数（如HS256、HS384或HS512）来生成的。这样，接收方可以使用相同的算法和密钥来重新计算签名，并与原始签名进行比较，以验证数据的完整性和真实性。 对头部和载荷部分进行加密计算而来
jwt生成

不需要记生成代码，因为在公司实际操作中是使用提供好的工具类如 jwtUtil，直接调用即可。

注意事项：

  • JWT校验时使用的签名秘钥和算法，必须和生成JWT令牌时使用的秘钥和算法配套
  • 如果JWT令牌解析校验时报错，则说明jwt令牌被篡改或失效了，令牌非法

拦截器是Spring框架中的一个概念，类似于Servlet中的过滤器，主要用于拦截用户请求并做出响应。****

  • 在拦截器类里编写具体拦截的代码
  • 在配置类中将该拦截器类注册并选择不拦截哪些类（比如登录接口、注册接口）

拦截器常重写的方法：

  • preHandle: 请求到达接口之前执行
  • afterCompletion: 请求(接口)结束后执行

注意事项：

  • 使用拦截器统一验证令牌
  • 登录和注册接口不拦截
registry.addInterceptor(loginInterceptor).excludePathPatterns("/user/login", "/user/register");  // 不拦截登录和注册接口

单元测试是开发者编写的一小段代码，用于检验被测代码的一个很小的、很明确的功能是否正确。

单元测试的主要目标是发现代码中的缺陷，它是一种白盒测试，测试人员需要了解被测单元的内部逻辑和工作原理。单元测试的执行速度较快，可以快速定位代码中的问题，并且能够覆盖代码的各个分支和路径。

  • 在maven工程中，单元测试代码通常写在test包下

Get请求和Post请求

两者的地址状态栏和请求实体对比：

  • get方式

  • post方式

1，区别：

（1）Get请求的数据（参数）会显示在地址栏，而Post不会，所以，Post比Get更加安全。

（2）Post请求的参数存放到了请求实体中，而Get没有请求实体，Get是存储在请求行中。

（3）数据传输Post有优势：Get方式请求的数据不能超过2k，而Post 没有上限。

（4）浏览缓存Get有优势：Get具有数据缓存，而Post没有。

✿ 从优势角度看，数据传输使用Post，数据浏览查询使用Get。即查询时使用Get，其他时候使用Post。

ThreadLocal

提供线程局部变量

  • 用来存取数据: set()/get()
  • 使用ThreadLocal存储的数据，线程安全（达到了线程的隔离）
  • 用完记得调用remove方法释放

用处：比如使用全局的ThreadLoca，可以在拦截器中往ThreadLocal中存入业务数据，这样其他接口get数据的时候直接从ThreadLocal中get即可，也是一般用提供的工具类 ThreadLocalUtil

动态sql

动态sql一般使用映射配置文件的方式来编写，该文件需要放到Java项目的src/main/resources目录下

  • 该配置文件的URL要对应于其生效的mapper类的URL，如其生效于位于java包下的org/example/mapper/ArticleMapper, 则该配置文件为resource包下的/org/example/mapper/ArticleMapper.xml

对于组合条件查询来说动态sql很有用比如：

<!--动态sql-->
<select id="list" resultType="org.example.mapper.ArticleMapper">
    select * from article
    <where>
        <if test="categoryId!=null">
            category_id = #{categoryId}
        </if>
        <if test="state!=null">
            and state = #{state}
        </if>
        and create_user=#{userId}
    </where>
</select>
